## 🚀 القسم الأول: دليل التشغيل السريع

### ١. المتطلبات الأساسية

| المكوّن | الإصدار المطلوب | الوظيفة |
|---------|----------------|---------|
| Node.js | v22+ | محرك التنفيذ الأساسي |
| pnpm | v9+ | مدير الحزم للـ Monorepo |
| MongoDB | v7+ | قاعدة البيانات المحلية |
| LM Studio | أحدث إصدار | محرك الذكاء الاصطناعي المحلي |
| Electron | v28+ | غلاف نظام التشغيل |

### ٢. تشغيل النظام

```bash
# تثبيت جميع الحزم
pnpm install

# تشغيل خادم التطوير (الواجهة الأمامية)
pnpm --filter nexus-desktop dev

# تشغيل النواة الخلفية
pnpm --filter backend-core dev
```

### ٣. اتصال قاعدة البيانات

يعمل النظام مع MongoDB محلي على المنفذ `27017`. قاعدة البيانات الرئيسية: `NexusOS`.

**المجموعات الأساسية:**
- `ai_request_cache` — ذاكرة التخزين المؤقت للطلبات الذكية
- `EncryptedVault` — الخزنة المشفرة بتقنية AES-256-GCM
- `profiles` — ملفات المستخدمين المشفرة

### ٤. إعداد LM Studio

يتطلب النظام تشغيل خادم LM Studio على منفذين:

| المنفذ | النموذج | الاستخدام |
|--------|---------|-----------|
| `1234` | نموذج 7B (سريع) | واجهة المستخدم، الترجمة، التصميم |
| `1235` | نموذج 14B (عميق) | المنطق المعقد، الهندسة المعمارية، التحليل الأمني |

**التوجيه الذكي:** يقوم `nexusPrimeEngine` بتصنيف كل مهمة تلقائياً إلى `FAST_UI` أو `SMART_LOGIC` ويختار النموذج المناسب.

## 🧠 القسم الثاني: النواة المركزية للذكاء الاصطناعي

### ١. المنسق الذكي (AiOrchestrator)

**الملف:** `apps/backend-core/src/services/aiOrchestrator.js`

المنسق الذكي هو البوابة المركزية لجميع طلبات الذكاء الاصطناعي في نيكسوس. يعمل بنظام **طوابير الأولوية الثلاثية:**

| الأولوية | الرمز | الاستخدام |
|-----------|-------|-----------|
| حرجة | `CRITICAL` | إصلاح الأعطال، الأمان |
| عالية | `HIGH` | الوكلاء الفرعيون (OpenClaw) |
| منخفضة | `LOW` | المحادثات العادية |

**آلية التخزين المؤقت الصفري:**
كل طلب يُجزّأ بخوارزمية SHA-256. إذا تطابقت البصمة مع طلب سابق، يُعاد الرد فوراً دون استهلاك موارد VRAM.

### ٢. محرك نيكسوس-برايم (NexusPrimeEngine)

**الملف:** `apps/backend-core/src/services/nexusPrimeEngine.js`

المحرك الأساسي الذي يدير:
- **التوجيه الذكي للنماذج** — اختيار تلقائي بين نموذج 7B و 14B
- **إدارة ذاكرة VRAM** — تحميل وتفريغ النماذج حسب الحاجة
- **الحلقة العاملة (Agentic Loop)** — تنفيذ الأدوات المحلية تلقائياً
- **تحليل علامات التفكير** — فصل `<think>` عن الاستجابة النهائية

### ٣. الأدوات المحلية المتاحة

| الأداة | الوظيفة | مستوى الأمان |
|--------|---------|-------------|
| `readFile` | قراءة ملفات المشروع | آمن — داخل حدود المشروع فقط |
| `writeFile` | كتابة ملفات (مع نظام الموافقة) | مُراقب — يتطلب موافقة |
| `executeCommand` | تنفيذ أوامر النظام | محمي — بوابة الحجر الأمني |
| `scrapeWebPage` | استخراج محتوى الويب | مقيد |
| `executeCode` | تشغيل كود في بيئة معزولة | محمي — Docker Sandbox |

## 🦞 القسم الثالث: نظام الوكلاء الفرعيين (OpenClaw)

### ١. نظرة عامة

تم استيعاب نظام OpenClaw في المرحلة 53 وتطويره عبر المراحل 55-56 ليصبح جزءاً أصيلاً من نيكسوس. الوكلاء الفرعيون قادرون على:

- ✅ تنفيذ أوامر النظام المحلية تلقائياً
- ✅ قراءة وتحليل الملفات
- ✅ توليد استجابات ذكية مع نتائج فعلية
- 🛡️ خضوع كامل لبوابة الحجر الأمني

### ٢. خط الأنابيب الكامل (God-Mode Pipeline)

```
المستخدم → المنسق الذكي → النموذج المحلي 14B
    ↓
النموذج يولّد <tool_call> JSON
    ↓
المنسق يعترض ويستخرج الأداة المطلوبة
    ↓
_executeTool → تنفيذ محلي مع حراسة أمنية
    ↓
النتيجة تُعاد للنموذج → تجميع الإجابة النهائية
```

### ٣. جسر الربط (OpenClawBridge)

**الملف:** `apps/backend-core/src/services/openClawBridge.js`

- `autoUpdateCheck()` — فحص تحديثات OpenClaw من GitHub
- `spawnLocalSubagent(task)` — إطلاق وكيل فرعي لتنفيذ مهمة محددة

## 🛡️ القسم الرابع: بروتوكولات الحماية

### ١. بروتوكول انعدام الثقة (Zero-Trust Vault)

**خدمة التشفير:** `apps/backend-core/src/services/cryptographyService.js`

- **الخوارزمية:** AES-256-GCM (تشفير عسكري)
- **المفتاح الرئيسي:** مخزن في `.env.local` (خارج Git)
- **آلية العمل:**
  1. مسح جميع البيانات الحساسة في المشروع
  2. تشفير كل ملف بمفتاح فريد
  3. تخزين النص المشفر في MongoDB (`EncryptedVault`)
  4. استبدال الملف الأصلي بمرجع أمني

### ٢. الحارس الحراري (Thermal Sentinel)

- مراقبة درجة حرارة وحدة المعالجة المركزية
- إبطاء الاستدلال تلقائياً عند تجاوز العتبات الحرارية
- إشعارات في الوقت الفعلي عبر واجهة المستخدم

### ٣. غرفة معادلة الضغط (Nexus Airlock)

نظام مراجعة التغييرات قبل حقنها في قاعدة الكود:
- كل تعديل يُقترح كـ "رقعة معلقة" (Pending Patch)
- يتطلب موافقة بشرية قبل التطبيق
- سجل كامل للتدقيق في `NEXUS_INTEGRITY_LEDGER.json`

### ٤. بوابة الحجر الأمني (OpenClaw Quarantine Gate)

**المرحلة 56 — مُستوعَبة من OpenClaw:**

ثلاث طبقات دفاعية في `_toolExecuteCommand`:

1. **القائمة السوداء الموسعة** — 20+ نمط خطير (`curl`, `wget`, `dd`, `chmod 777`, `powershell -enc`...)
2. **حراسة حقن الأوامر** — حظر `$(...)`, علامات التنصيص العكسية, سلاسل الأنابيب
3. **منع هروب المسار** — حظر `../` خارج جذر المشروع
